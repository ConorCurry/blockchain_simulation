Blockchain Simulation

Conor Curry, Robbie McKinstry, & Natalie Boehm

Motivation

New problems have arisen leading some to doubt long term feasability of Bitcoin currency and underlying blockchain technology as the protocol becomes more accepted

Lots of discussion surrounding 'blocksize debate' - 1 MB limit causes delays in low-fee transactions, leading to larger fee amounts in order to be put on the blockchain in a reasonable of time

Runs of full blocks can last for several hours, increasing the likelihood of security breaches

When coinbase transactions stop generating Bitcoin, fees will be relied upon to support miners in verification of blocks - it is unknown how scarce a resource space on the blockchain will be, or how competitive the 'fee market' will become

Specific Problem Statement

We seek to figure out what impact changes in the blockchain block size have on transaction fee amounts, transaction times, and the Bitcoin ecosystem in general

Approach

A diagram of the blockchain

Transaction data portion of block contains multiple transactions. Merkle root represents a singular hash of each transaction copy.

4 Phases

  1. Collect public information from the blockchain
  2. Model data collected
  3. Generated simulation based on model generated
  4. Validate model data against empirical data

Data Collection

  • Used exposed API on blockchain.info to collect transaction information, using a collection script to collect metadata for 10,000 contiguous blocks spanning the course of a week
  • Calculated acceptible number of requests per minute
  • When executing script, function was included that would sleep in main thread if number of requests per minute exceeded rate
  • Parsed JSON respons and transformed data into CSV format before storing data results on disk
  • For each block, only interested in five attributes:
    1. Number of transactions
    2. Fee cost
    3. Time at which block was added to blockchain
    4. Block size
    5. Unique identifer hash

Input Modeling

  • Modeled interarrival times of each block, normalized the data, and ran the Kolmogrov-Smirnov test on the resulting data

Simulation

  • Arrival times generated for each block
  • Lambda value generated for arrival times of transactions per block
  • Run two seperate simulations, one as the control group using the original block size of 1 MB, the other being the experimental group using a doubled block size
  • For the experimental group, test if the new block size can saturate the number of incoming transactions per rate time

Model Validation

  • Compare control group data to actual data previously collected
  • Compare results of control group to those of the experimental group

Experimental Setup

Goal: To conduct an analysis of the stability of the queueing system found in Bitcoin

Given parameters determined via input validated data from the blockchain itself, we seek to determine the likelihood of runs of full blocks, likely cost estimates per byte on the blockchain, and the possible effects of increasing the block size

We hope to observe the dynamics of user-miner interactions in the Bitcoin system, as a function of inclusion in the blockchain

Setup

Used blockchain.info API and Python libraries scipy, numpy, and matplotlib to collect, model, simulate, and validate data
